# 信号与置信度

## 信号类型

`vite-browser` 会组合几类运行时证据：

- HMR 时间线
- 模块图快照和追踪
- 当前浏览器运行时错误
- 框架组件检查结果
- store 变化和 changed keys
- 日志、网络请求和页面侧 eval 输出

这些信号的权重并不完全一样。通常当前错误和最近 HMR 更新是最强锚点，而渲染路径或 store 传播线索更依赖时序和上下文。

## 如何读关联输出

### `correlate errors`

当你的问题是下面这种时，优先看它：

```text
现在看到的错误，最可能和哪次最近更新有关？
```

它在这些条件下最强：

- 复现刚刚发生
- 当前错误栈还在
- 更新时间窗不大

### `correlate renders`

当你的问题是下面这种时，优先看它：

```text
最近一次状态或模块变化，是否很可能沿着某条重渲染路径传到了当前问题？
```

它应该被看作缩小范围的证据，而不是框架内部严格回放。

### `diagnose propagation`

当你想要一个基于现有 store、render、error 证据的规则化总结时，使用它。

如果证据链不完整，它会刻意保持保守。

## 置信度建议

### High confidence

可以把它当成强排查线索。它依然不是数学证明，但通常足以决定下一步应该优先看哪个文件或哪次复现。

### Medium / plausible

适合收缩怀疑范围，不适合直接当成根因结论。

### Weak or absent evidence

通常意味着你应该重新复现、缩短时间窗，或者切到日志、网络等相邻信号继续看。

## 最佳实践

- 需要源码上下文时，优先从 `errors --mapped --inline-source` 开始
- 实时复现时优先用短时间窗，比如 `5000` ms
- reload 或导航之后重新跑命令，让当前运行时视图保持新鲜
- 让最强的信号决定下一步，而不是机械地把所有命令都跑一遍
